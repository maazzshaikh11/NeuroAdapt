Feature Ticket List
Project: NeuroAdapt
Source PRD: NeuroAdapt Product Requirements Document, Team DSY, TechForGood 2026.
This ticket list breaks the MVP into buildable tasks that can be used directly by an AI coding tool or engineering team. It follows the PRD scope: Chrome extension first, in-place text simplification, Bionic Reading mode, Focus Mode, JWT auth, MongoDB profile sync, and an LLM-backed backend. The MVP is meant to be demonstrable in a single live session with a clear before/after transformation.
Priority Legend
Must-have for launch = required for the Month 1 MVP
Should-have = important, but can ship after the first demo
Nice-to-have = later release only

1. Design System Overview

NeuroAdapt's design system is built on a single principle: the interface must never become the problem it is trying to solve. Every visual decision — colour, spacing, type size, component shape — is chosen to reduce cognitive load, not add to it. This document governs every pixel of the extension popup, the web dashboard, and all injected UI elements.

🎯 Design Philosophy
Clarity over cleverness — no decorative UI that adds noise.
Contrast first — every element meets WCAG 2.1 AA contrast (minimum 4.5:1).
Consistent rhythm — spacing follows an 8px base grid, no exceptions.
Neurodivergent-safe — no flashing animations, no auto-playing media, no infinite scroll.
Dark/light adaptable — every colour token has a light and dark mode equivalent.

3. Typography System

NeuroAdapt uses a two-family type system. Inter is the UI font — designed for screens and readability at small sizes. OpenDyslexic is the user-selectable accessibility font — users with Dyslexia can switch to it globally via the extension toggle. Both are loaded as web fonts; Inter via Google Fonts, OpenDyslexic via a self-hosted CDN fallback.

1. Product Overview
1.1 What Is NeuroAdapt?
NeuroAdapt is a browser extension and cloud-synced dashboard that acts as an intelligent cognitive accessibility layer on top of any website. It uses AI-powered Natural Language Processing to dynamically simplify, restructure, and declutter web content in real-time — making the web genuinely usable for neurodivergent individuals, elderly users, and first-generation internet users.

Unlike static reading aids or screen readers, NeuroAdapt does not require users to leave the page they are on. It works in-place, transforming the content they are already viewing.

1.2 The Problem It Solves
Core Problem Statement
Over 15–20% of the global population is neurodivergent, yet the modern internet is built for neurotypical, highly literate users. High text density, complex jargon, cluttered interfaces, and overwhelming visual noise create a digital divide that existing accessibility tools (screen readers, contrast checkers) entirely fail to address.

Current solutions fail in three specific ways:
Static tools require copy-pasting into third-party apps — breaking the reading flow entirely.
W3C WCAG guidelines focus on physical disabilities (contrast, ARIA labels) and offer nothing for cognitive load.
Tools like Grammarly help users write better — but nothing helps users read and comprehend better in real-time.

1. Authentication Strategy

Authentication is the security gate that verifies a user is who they claim to be before granting access to NeuroAdapt. Below is the complete strategy — what method we use, why we chose it, and exactly how it works.

1.1  Chosen Method: JWT + Email/Password + OAuth
NeuroAdapt uses a two-track login system:
Email & Password — for users who prefer traditional sign-up (teachers, institutions).
Google OAuth 2.0 — one-click login for individual users (students, elderly users).
Both tracks issue a JWT (JSON Web Token) upon success. Think of a JWT like a temporary, tamper-proof ID badge. Once logged in, the user carries this badge on every request to prove their identity — no need to re-enter a password on each action.

💡 Why JWT? 
JWTs are stateless — the server doesn't need to store session data in a database. This means NeuroAdapt can scale from 100 to 100,000 users without performance degradation.

1.3  Secure Storage Rules
Access Token → stored in memory (JavaScript variable). Never in localStorage. Reason: localStorage is accessible to any script on the page — a major XSS attack risk.
Refresh Token → stored in an HttpOnly cookie. This type of cookie cannot be read by JavaScript at all — only the browser can see it, making theft nearly impossible.
Passwords → never stored as plain text. We use bcrypt with a cost factor of 12, which means even if our database were leaked, passwords would take years to crack.

3. Row-Level Security (Database Rules)

Row-Level Security (RLS) means that even if a user successfully logs in, they can only access database records that belong to them — not anyone else's. Think of it like a hotel key card: it opens only your room, not every room on the floor.
NeuroAdapt uses MongoDB. Since MongoDB doesn't have built-in RLS like PostgreSQL, we enforce it at the backend code level — every database query is automatically filtered by the authenticated user's ID.

🔒 Core Security Rule 
Every MongoDB query that touches user data MUST include { userId: req.user.id } as a filter condition. This single rule prevents the most common data-leakage bugs. It must be enforced in code review.

5. Pre-Launch Edge Cases

Edge cases are unusual or unexpected situations that can break the app or create security vulnerabilities if not explicitly handled. Every item below must be resolved before the NeuroAdapt prototype is demonstrated publicly.

5.1  Authentication Edge Cases
🔄  Concurrent Login on Multiple Devices  [ HIGH ]
⚠ Problem: User logs in on Chrome and Firefox simultaneously. Old refresh token should not break the new session.
✅ Fix: Issue refresh tokens per-device with a device fingerprint. Store up to 5 active sessions per user in MongoDB. Revoking one does not affect others.

📧  Email Changed but Old Token Still Valid  [ HIGH ]
⚠ Problem: User changes email, but their old JWT (valid for 15 min) still works with the old email claim.
✅ Fix: Embed an email_version counter in the JWT. Increment it on email change. Backend rejects any token where the counter doesn't match.

🔗  Google OAuth Account Collision  [ MEDIUM ]
⚠ Problem: User signs up with email/password first, then tries to log in via Google with the same email.
✅ Fix: On OAuth login, check if the email already exists. If yes, automatically link accounts and redirect to dashboard with a notice: 'Your Google account has been linked.'

⏱️  Race Condition on Token Refresh  [ MEDIUM ]
⚠ Problem: Two API calls happen simultaneously with an expired access token. Both try to refresh — one succeeds, one creates a new orphaned token.
✅ Fix: Implement a refresh mutex: if a refresh is already in-flight, queue subsequent requests and serve them the new token once it arrives. Discard duplicate refresh attempts.

5.2  AI Feature Edge Cases
📝  User Selects DOM Elements, Not Text  [ HIGH ]
⚠ Problem: When the extension tries to read the DOM, it might grab navigation menus, cookie banners, or hidden HTML instead of article text.
✅ Fix: Filter selected content: strip any element with role='navigation', aria-hidden='true', or class names containing 'nav', 'menu', 'footer', 'cookie'. Only pass visible body text to the AI.

🔁  Repeated Identical Simplification Requests  [ HIGH ]
⚠ Problem: The same paragraph gets simplified 1,000 times — wastes AI API budget rapidly.
✅ Fix: Cache the simplified result by SHA-256 hash of the input text. Check cache before calling the AI API. Cache expires in 7 days. Estimated 60-70% cost reduction.

🌐  AI Returns Text in Wrong Language  [ MEDIUM ]
⚠ Problem: If a webpage is in Hindi and the AI is prompted in English, it may return a mix of languages.
✅ Fix: Include the detected language in the prompt: 'Simplify the following [detected_lang] text, and respond only in [detected_lang].' Use the browser's navigator.language API to detect.

💥  AI Injects Malicious HTML in Response  [ CRITICAL ]
⚠ Problem: A poorly constrained AI response could return HTML/JS tags if the original webpage text contains adversarial content (prompt injection).
✅ Fix: Sanitize all AI responses with DOMPurify before injecting into the DOM. Strip all script tags, event handlers (onclick, onerror), and iframe elements from AI output.

5.3  Browser Extension Edge Cases
🔒  Extension Runs on Banking / Sensitive Sites  [ CRITICAL ]
⚠ Problem: The extension has DOM access — users might run it on their banking portal, exposing sensitive data to API calls.
✅ Fix: Add a default blocklist: extension does not activate on domains like *.bank.*, *.gov.in, mail.*, etc. Show a warning banner and disable the 'Simplify' button on blocked sites.

📄  Single-Page App (SPA) Navigation  [ HIGH ]
⚠ Problem: Websites built with React/Angular change content without reloading the page. The extension may show stale simplified text after navigation.
✅ Fix: Use a MutationObserver to detect DOM changes. On significant DOM mutation, reset simplified content and re-inject the NeuroAdapt widget.

🌑  Extension Conflicts with Website's Own Dark Mode  [ LOW ]
⚠ Problem: Applying NeuroAdapt's focus mode on a site that already has its own dark mode may create double-darkness or invisible text.
✅ Fix: Before applying styles, check document.body's computed background color. If luminance < 30%, skip background overrides. Offer a 'Light Mode Override' toggle in the extension popup.

5.4  Data & Privacy Edge Cases
🗑️  User Requests Account Deletion (DPDP Compliance)  [ HIGH ]
⚠ Problem: India's Digital Personal Data Protection Act 2023 requires that user data be deleted within a reasonable timeframe upon request.
✅ Fix: Build a DELETE /api/user/me endpoint. On call: mark account as deletion_pending, anonymize personal fields within 24 hours, permanently purge MongoDB records within 30 days. Send a confirmation email.

🔐  Admin Accidentally Assigned to New Account  [ HIGH ]
⚠ Problem: A typo in the database could accidentally grant a new user admin privileges.
✅ Fix: Admin role can only be assigned via a direct, authenticated CLI script run by a core team member — never via an API endpoint. All role changes are logged to auditLogs.

📊  Analytics Leak User Behaviour Patterns  [ MEDIUM ]
⚠ Problem: Aggregate analytics shown to admins could reveal private behaviour if the dataset is too small (e.g., only 1 user from a specific school).
✅ Fix: Apply k-anonymity: only display aggregate stats for groups of 5+ users. Suppress data points from groups smaller than 5.

1. Recommended Tech Stack with Reasoning
Every technology choice below was made with three constraints in mind: 1-month build time, zero infrastructure cost during the internship prototype phase, and a codebase that can scale to production without a full rewrite.

1.1 Frontend — React.js + Tailwind CSS
Aspect
Detail
Technology
React.js (with Webpack/Vite for MV3 Chrome extension bundling)
Why React
Component-based architecture maps perfectly to the extension's popup panel, floating widget, and settings page — all as isolated reusable components. React's virtual DOM also prevents costly re-renders when injecting simplified text back into a live website.
Why not Vue/Svelte
React has vastly more Chrome extension examples, starter templates, and community support. The team's existing knowledge also reduces ramp-up time.
Styling
Tailwind CSS — utility-first, no context switching between CSS files and JSX. Critical for rapid UI iteration in a 1-month timeline.
Bundler
Vite with CRXJS plugin — purpose-built for Chrome Extension Manifest V3. Handles hot-module replacement during development, which standard CRA cannot do for extensions.

1.2 Content Script — Vanilla JavaScript
The DOM manipulation layer (Focus Mode, Bionic reading, text injection) is written in plain JavaScript — NOT React. Reason: content scripts are injected into third-party websites. Injecting a React runtime into every page would add 40–120KB of overhead per tab, causing noticeable slowdown. Vanilla JS keeps the injection under 8KB.

1.3 Backend — Node.js + Express.js
Aspect
Detail
Technology
Node.js v20 LTS + Express.js v4
Why Node
JavaScript on both frontend and backend means the team shares one language context — no context switching. Node's non-blocking I/O is ideal for the primary workload: forwarding text to an external LLM API and waiting for a response.
Why Express
Minimal, un-opinionated, and the most widely documented framework for Node. For a 4-route MVP, NestJS or Fastify would be overkill.
Auth
JSON Web Tokens (JWT) via jsonwebtoken package. Stateless auth means the backend does not need session storage, keeping the architecture simple.
Rate Limiting
express-rate-limit middleware — protects the LLM API key from abuse. Configured to 20 requests/minute per IP.

1.4 Database — MongoDB Atlas
Aspect
Detail
Technology
MongoDB Atlas M0 free tier + Mongoose ODM
Why MongoDB
User accessibility preferences are a loosely structured JSON blob that evolves frequently (new features add new fields). MongoDB's schema-less documents absorb these changes with zero migrations — critical in a fast-moving 1-month sprint.
Why not PostgreSQL
SQL's rigid schema would require ALTER TABLE migrations every time a new accessibility toggle is added. MongoDB's flexible documents eliminate this friction entirely during prototype phase.
Caching
Simplified text results are cached directly in MongoDB (SimplificationCache collection) using a SHA-256 hash of the input text as the lookup key. This avoids a Redis dependency in the prototype.

3. Database Schema
Schema Philosophy
NeuroAdapt uses three MongoDB collections. Each document is explained below in plain English — what it stores, why that field exists, and how collections relate to each other. 3.2 Collection: SimplificationCache
Stores the results of previous AI simplification calls. When a user simplifies text that another user has already simplified, NeuroAdapt serves the cached result instantly — no LLM API call needed. This is the primary cost-control mechanism.

// models/SimplificationCache.model.js
const CacheSchema = new Schema({

  // ── Lookup Key ────────────────────────────
  inputHash:   { type: String, required: true, unique: true, index: true },
  // SHA-256 of (originalText + simplificationLevel) — composite key

  // ── Content ───────────────────────────────
  originalText:    { type: String, required: true },
  simplifiedText:  { type: String, required: true },
  simplificationLevel: { type: String, enum: ['basic','standard','academic'] },

  // ── Performance Metadata ──────────────────
  hitCount:    { type: Number, default: 1 },   // How many times this cache was served
  llmProvider: { type: String },               // 'gemini' | 'openai' — for A/B quality tracking
  latencyMs:   { type: Number },               // How long the original LLM call took
  createdAt:   { type: Date, default: Date.now },
  lastHitAt:   { type: Date, default: Date.now },

  // TTL: Auto-delete cache entries after 30 days of no use
  expiresAt:   { type: Date, index: { expireAfterSeconds: 0 } },

}, { timestamps: false });

3.2.1 Field Explanations — SimplificationCache
Field
Type
Plain-English Purpose
inputHash
String
A SHA-256 fingerprint of the original text + simplification level combined. Acts as the cache key. If two users highlight the exact same paragraph and both want 'standard' simplification, inputHash is identical — cache hit.
hitCount
Number
Tracks how many times this specific simplification was served from cache rather than calling the LLM. High hitCount = high ROI on this cached entry. Used in the success metrics dashboard.
expiresAt
Date
MongoDB TTL index — automatically deletes cache entries 30 days after their last use. Prevents the cache collection from growing unboundedly and keeps costs predictable.
llmProvider
String
Records whether this result came from Gemini or OpenAI. Enables quality comparison between providers over time, informing which to default to.

3.3 Collection: UsageLogs
An append-only log of every simplification event. Never updated, only inserted. Used for the research paper's empirical data — measuring how many users used the tool, on which types of sites, and how often.

// models/UsageLog.model.js

5.3 Configuration Notes
Config Item
What You Must Do
MongoDB Atlas IP Whitelist
In Atlas → Network Access, add 0.0.0.0/0 during development. For production, restrict to Railway/Render's outbound IPs.
Chrome Extension ID
Load the unpacked extension in Chrome first (chrome://extensions → Developer Mode → Load Unpacked). Copy the Extension ID and paste it into ALLOWED_ORIGINS in the backend .env.
JWT_SECRET strength
Must be at least 32 random characters. Generate with: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
Gemini Free Tier Limit
15 requests/minute on the free tier. The rate limiter on the backend is set to 20 req/min per IP — lower than the LLM limit so you never accidentally exceed the API quota.
MongoDB Atlas Free Tier
M0 cluster has 512MB storage and 100 max connections. More than sufficient for the prototype. Connection string auto-closes idle connections after 10 minutes — handled by Mongoose.
OpenDyslexic Font
Host the OpenDyslexic .woff2 font file inside the extension's /public/fonts/ folder. Chrome MV3 CSP blocks loading external fonts from third-party CDNs in content scripts.
LLM Prompt Engineering
The system prompt in llm.service.js must explicitly instruct the model: 'Do not add any explanation or preamble. Return ONLY the simplified text.' Without this, the LLM adds filler like 'Here is the simplified version:' that breaks DOM injection.

6. End-to-End Data Flow (Simplification Request)
This traces one complete simplification request from the moment the user clicks 'Simplify' to the moment they see the transformed text on the page.

Step
Layer
What Happens
1
Content Script
User highlights text and clicks the floating widget. content.js captures window.getSelection().toString() and the hostname of the current page.
2
Content Script
Sends a chrome.runtime.sendMessage({ type: 'SIMPLIFY', text, level, hostname }) to the background service worker.
3
Background Worker
Retrieves the JWT from chrome.storage.local. Makes an HTTPS POST to /api/simplify with the token in the Authorization header.
4
Backend: Middleware
auth.middleware.js verifies the JWT. If invalid/expired, returns 401. rateLimit.middleware.js checks request count per IP — if exceeded, returns 429.
5
Backend: Cache
cache.service.js hashes (text + level) with SHA-256. Looks up the hash in SimplificationCache. CACHE HIT: returns simplifiedText immediately, increments hitCount. CACHE MISS: proceeds to Step 6.
6
Backend: LLM
llm.service.js constructs the system + user prompt, calls Gemini Flash API. Records latencyMs. Strips any LLM preamble from the response.
7
Backend: Save
Saves the result to SimplificationCache. Appends a UsageLog entry. Increments user.totalSimplifications counter.
8
Content Script
Receives { simplifiedText, cacheHit, latencyMs }. Finds the DOM node that contained the selected text. Applies a CSS fade-out, replaces innerHTML with simplifiedText, fades back in.
9
UI
A small 'Original / Simplified' toggle button appears below the transformed text, letting the user switch back if needed.
